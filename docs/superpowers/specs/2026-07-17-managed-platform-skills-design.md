# SkillsHub 管理的平台技能设计

## 目标

平台页面只展示由 SkillsHub 安装或由用户通过 SkillsHub 明确导入的技能。平台自身或其他工具写入平台技能目录的内容仍可被扫描器发现，但不得显示为 SkillsHub 安装、不得计入侧栏数量，也不得提供卸载操作。

## 数据模型

在 `skill_installations` 增加 `is_managed` 字段：

- SkillsHub 的安装、导入和备份恢复流程写入 `is_managed = 1`。
- 文件系统扫描首次发现的目录写入 `is_managed = 0`。
- 扫描更新已有记录时保留原有 `is_managed`，不能把外部技能升级为受管技能，也不能把受管技能降级。
- 旧数据库迁移时，仅将能够关联到中央技能或技能资源库规范路径的安装记录标记为受管；无法可靠证明来源的记录保持非受管，避免误删平台自己的技能。

## 查询与行为

- 平台技能列表只查询 `is_managed = 1` 的安装记录。
- 技能详情中的平台安装状态只使用受管记录。
- 侧栏平台数量只统计受管记录。
- 卸载、批量操作和备份只处理受管记录。
- 扫描器仍维护非受管观察记录，以便处理目录变化，但这些记录不进入用户可管理界面。
- 用户随后通过 SkillsHub 安装同名技能时，安装操作将对应记录升级为受管记录。

## 安全边界

- 迁移和扫描均不删除外部技能文件。
- 非受管记录不能触发卸载或覆盖。
- 现有中央技能和资源库技能的安装关系继续保留。
- Codex 的安装目录保持 `~/.codex/skills`，不会重新与中央目录合并。

## 验证

- 扫描 Codex 自带实体技能后，平台列表和侧栏数量均不包含它。
- 通过 SkillsHub 安装到 Codex 后，该技能正常显示并可卸载。
- 重新扫描不会改变受管标识。
- 外部技能与受管技能同目录共存时，仅显示受管技能。
- 完整 Rust 测试、Clippy 和前端生产构建通过。
